## Java 21：最具影响力的 LTS

Java 21 是继 Java 17 之后最重要的 LTS 版本，引入的特性对中间件和后台服务开发影响深远。

### 虚拟线程（Virtual Threads）

虚拟线程是 Java 21 最受瞩目的特性。它解决了"每请求一线程"模型中线程池大小受限的问题：

```java
// 传统线程池
var executor = Executors.newFixedThreadPool(200);

// 虚拟线程版本
var executor = Executors.newVirtualThreadPerTaskExecutor();
```

虚拟线程的调度由 JVM 管理，不再依赖操作系统线程。这意味着你可以创建数十万个虚拟线程，而不会耗尽系统资源。对于中间件开发来说，这尤其有价值——网关、代理层、消息队列消费者等 IO 密集型场景，虚拟线程能大幅简化并发模型。

以一个简单的代理服务器为例：

```java
public class SimpleProxy {
    public static void main(String[] args) throws Exception {
        var server = new ServerSocket(8080);
        while (true) {
            var client = server.accept();
            Thread.ofVirtual().start(() -> handle(client));
        }
    }

    static void handle(Socket client) {
        try (client) {
            var upstream = new Socket("upstream", 8081);
            // 双向转发
            Thread.ofVirtual().start(() -> transfer(client, upstream));
            transfer(upstream, client);
        } catch (IOException e) {
            // 日志处理
        }
    }
}
```

**使用虚拟线程的关键注意事项**：

- **不要池化**：虚拟线程的设计意图就是用完即弃，池化反而浪费
- **synchronized 会钉住线程**：虚拟线程内部使用 synchronized 时会被钉在载体线程上，优先改用 ReentrantLock
- **ThreadLocal 成本变高**：大量虚拟线程意味着大量 ThreadLocal，需要谨慎使用

### 模式匹配（Pattern Matching）

switch 表达式在 Java 21 中获得了模式匹配能力，配合 sealed class 可以实现非常优雅的分支处理：

```java
public sealed interface Result
    permits Success, Failure, Timeout {}

public record Success<T>(T data, long latency) implements Result {}
public record Failure(String message, int code) implements Result {}
public record Timeout(int timeoutMs) implements Result {}

public String handle(Result result) {
    return switch (result) {
        case Success(var data, var latency) ->
            "成功: " + data + " (耗时: " + latency + "ms)";
        case Failure(var msg, var code) ->
            "失败: " + msg + " (代码: " + code + ")";
        case Timeout(var ms) ->
            "超时: " + ms + "ms";
    };
}
```

这种模式在中间件开发中特别有用——协议解析、状态机、路由分发等场景都大量使用类型层级判断。

### Record Pattern

Record Pattern 让你在解构 record 时可以嵌套匹配，处理树形结构非常顺手：

```java
public record Request(String method, String path, Headers headers) {}
public record Headers(String contentType, int contentLength) {}

if (request instanceof Request(var method, var path, Headers(var type, var len))) {
    System.out.println(method + " " + path + " [" + type + "]");
}
```

### 字符串模板（String Templates）

Java 21 以预览特性引入了字符串模板，比字符串拼接和格式化都更直观：

```java
var name = "Shanhai";
var age = 30;
var msg = STR."\{name} is \{age} years old";
// 输出: Shanhai is 30 years old
```

在中间件开发中，拼接 SQL、构建 HTTP 请求、生成日志行等场景都能用上。

## Java 22：持续演进

Java 22 在 21 的基础上继续推进，值得关注的变化：

### 未命名变量和模式

用 `_` 替代不使用的变量名，消除编译警告：

```java
// 之前
try { var result = process(); } catch (Exception e) { log.error("失败"); }

// 之后
try { var _ = process(); } catch (Exception _) { log.error("失败"); }
```

在中间件开发中，这种简化很实用——比如只需要拿到结果但不需要变量名的时候。

### Stream Gatherers（预览）

Stream API 引入了一个新的中间操作 `gather`，支持自定义的中间操作，比之前的 `map`、`filter` 组合更灵活：

```java
var result = items.stream()
    .gather(myCustomGatherer())
    .toList();
```

对于消息批处理、数据聚合等场景，Gatherers 提供了更强的表达能力。

### 外部函数与内存 API（第三预览版）

FFM API 持续完善，未来 Java 可以不依赖 JNI 调用原生库。对于需要底层操作的中间件来说，这打开了新的可能性。

## 对中间件开发的影响

这些新特性对中间件开发的具体影响：

| 特性 | 应用场景 | 带来的价值 |
|------|----------|------------|
| 虚拟线程 | 网关、代理、RPC 框架 | 简化并发模型，提升吞吐量 |
| 模式匹配 | 协议解析、状态机 | 代码更可读，减少 instanceof 判断 |
| Record Pattern | 路由配置、请求解析 | 声明式数据转换 |
| 字符串模板 | SQL 构建、日志输出 | 避免拼接错误和注入风险 |

## 迁移建议

如果你的项目还在 Java 11 或 17，建议有计划地升级：

1. **先升级到 Java 21 LTS**，这步收益最大
2. **启用虚拟线程**，从 IO 密集的模块开始替换
3. **逐步引入模式匹配**，在新增代码中使用 switch 表达式
4. **关注上游依赖兼容性**，Spring Boot 3.x、Netty 等主流框架已全面支持 Java 21

升级不是一蹴而就的事，但 Java 21 带来的改进值得投入精力。
