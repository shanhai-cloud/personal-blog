## 什么是 Vibe Coding

2025 年 2 月，Andrej Karpathy 首次提出了 **Vibe Coding** 这个概念：

> "你完全沉浸于氛围中，拥抱指数级增长，甚至忘记代码的存在。这不是真正的编程——我只是看看东西、说说东西、运行东西，然后复制粘贴，而且它大多都能工作。"

简单说，Vibe Coding 就是用自然语言描述需求，让 AI 生成代码，开发者从"写代码"转向"审代码"。

## 工具生态

到 2026 年，Vibe Coding 的工具链已经相当成熟：

| 工具 | 定位 | 适用场景 |
|------|------|----------|
| **Cursor** | AI-native IDE | 日常开发，Composer 模式处理复杂任务 |
| **GitHub Copilot** | IDE 插件 | 补全和 inline 建议，轻量辅助 |
| **Claude Code** | 终端 Agent | 大规模重构、跨文件改动 |
| **Devin** | 自主 AI 工程师 | 独立完成任务，CI 集成 |
| **TRAE** | 国内 IDE | 中文需求理解更准确 |

## Cursor 的实际效果

Cursor 在 2026 年初公布的数据揭示了几个有意思的趋势：

**AI 生成代码的存活率在上升**——从年初的 76% 提升到 81%。这意味着 AI 代码的质量在持续改善，开发者越来越信任 AI 的输出。

**每次提交的代码量增长了 2.5 倍**，"超大型提交"（超过 1000 行）越来越常见。这改变了传统的代码审查节奏——逐行 Review 变得不现实，开发者更需要从架构层面做判断。

**头部效应显著**——P99 开发者产出的代码量是中位数用户的 46 倍。AI 放大的是有经验者的优势：知道怎么描述问题的人，得到的答案质量远高于不会提问的人。

## 实践中的三个原则

### 1. 拆解到足够小

AI 擅长完成明确、边界清晰的任务。把需求拆到够细，每个 Prompt 只做一件事：

```text
❌ "帮我写个用户管理系统"
✅ "为这个 User 实体添加一个根据邮箱查询用户的 JPA Repository 方法"
```

任务越小，AI 的成功率越高，出错时也更容易定位。

### 2. 用规则文件约束风格

在项目中维护一份规则文件（如 CLAUDE.md、AGENTS.md），写清楚代码风格、项目结构和约定。大多数 AI 工具都会读取这些文件作为上下文：

```markdown
# 项目规范

- 使用 Java 21，虚拟线程
- Controller 层统一使用 @RestController
- 异常使用全局 @ControllerAdvice 处理
- 数据库查询使用 QueryDSL
```

一份好的规则文件能显著减少 AI 的"风格漂移"。

### 3. 建立验证闭环

AI 生成的代码必须经过验证。几个有效的检查手段：

- **编译检查**：AI 经常写错 imports 和方法签名，第一关就要卡住
- **静态分析**：SonarQube、SpotBugs 等工具能发现 AI 代码中的常见模式
- **测试覆盖**：让 AI 自己为生成的代码写测试，然后运行验证

## 陷阱与反思

Vibe Coding 不是银弹。几个值得注意的问题：

**缺乏全局架构感**——AI 擅长让单个模块跑通，但很难设计出整体架构。它更倾向于在现有代码上"打补丁"而不是重构。

**技术债积累快**——AI 生成代码的可读性普遍偏低，变量命名、日志、错误处理经常是敷衍的。上线前需要人工清理。

**没有长期记忆**——AI Agent 每次对话都是全新的上下文，它会反复犯同一个错误，除非你把规则写进文件。

**安全风险**——AI 可能生成存在已知漏洞的代码（如 SQL 注入），安全审查不能省略。

有位资深工程师总结得很到位：**"Vibe Coding 是放大镜，它会放大经验、判断力和工程素养，也会放大认知缺失和方法论漏洞。"**

## VibeTensor 的启示

2026 年初，英伟达工程师许冰发布了 **VibeTensor**——一个完全由 AI 生成的深度学习框架，16 万行代码没有一行人类手写。这个项目的结论很有参考价值：

- AI 能写出功能正确的代码（所有测试通过）
- 但全局性能比 PyTorch 慢 1.7–6.2 倍
- 每个子系统的逻辑都对，拼在一起却有问题（弗兰肯斯坦效应）

## 总结

Vibe Coding 不是"不用学编程了"，而是**编程的形态变了**——从手写每一行代码，变成用自然语言描述意图、审查 AI 输出、在关键节点做架构决策。

限制你生产力的最终瓶颈，从"写代码的速度"变成了"定义问题的清晰度"。
