## 什么时候应该拆

微服务不是银弹。在决定拆分之前，先问自己三个问题：

1. **业务复杂度是否真的需要？**——团队人数少于 10 人、业务逻辑不复杂时，单体加模块化架构可能更合适
2. **拆分收益是否大于成本？**——网络延迟、数据一致性、运维复杂度都是代价
3. **是否具备基础设施？**——CI/CD、容器化、监控体系是微服务的前提条件

## 服务拆分原则

### 按业务边界拆分

核心原则是**一个服务只做一件事**。参考领域驱动设计（DDD）的限界上下文（Bounded Context）：

```
用户服务（User Service）      → 注册、登录、权限
商品服务（Product Service）    → 商品 CRUD、库存
订单服务（Order Service）      → 订单创建、状态流转
支付服务（Payment Service）    → 支付对接、退款
通知服务（Notification Service）→ 邮件、短信、推送
```

### 拆分步骤

1. **先单体后拆分**：先在一个项目里用模块组织代码，边界清晰后再拆成独立服务
2. **从边缘开始**：先拆独立性强、耦合度低的模块（如通知服务），最后拆核心业务
3. **数据独立**：每个服务拥有自己的数据库，避免跨服务 join

```java
// 错误做法：服务 A 直接查服务 B 的数据库
// 正确做法：服务 A 调用服务 B 的 API
```

## 服务间通信

拆完之后，服务之间怎么说话？两种主流方式：

### REST / gRPC（同步）

适用于请求-响应模式的实时查询：

```java
// 基于 Spring 6 的 HttpExchange
@HttpExchange("/users")
public interface UserClient {
    @GetExchange("/{id}")
    User getUser(@PathVariable Long id);
}
```

### 消息队列（异步）

适用于事件通知、解耦场景：

```
订单服务 → 发送 "order.created" 事件 → RabbitMQ/Kafka
                                      → 通知服务消费：发邮件
                                      → 库存服务消费：扣库存
```

## 数据一致性

微服务最棘手的问题：**怎么保证跨服务的数据一致？**

- **最终一致性**：大部分场景下接受短暂不一致，通过事件重试补偿
- **Saga 模式**：每个操作都有对应的补偿操作，出错了回滚
- **TCC（Try-Confirm-Cancel）**：两阶段补偿，适合资金类场景
- **Seata AT 模式**：通过代理 JDBC 数据源实现分布式事务，对业务代码侵入最小

## 总结

微服务的核心不是技术，而是**组织管理和架构治理**。技术选型随时可以换，但服务边界划分错了重构成本极高。

对于初学者来说，推荐的学习路径是：

1. 先写好一个单体项目
2. 理解模块化设计
3. 拆分一个非核心模块成独立服务
4. 逐步引入注册中心、网关、配置中心等基础设施
